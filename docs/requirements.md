## 1. 目的
ユーザーのPC作業内容（使用アプリ、閲覧サイト、バックグラウンド状況）を定期的かつ自動的に記録し、後から「何に時間を使ったか」を詳細に分析可能にする。

## 2. 機能要件

### アクティブウィンドウ監視
現在最前面にあるウィンドウの「アプリケーション名」と「ウィンドウタイトル」を取得する。

### ブラウザ（Google Chrome）詳細監視
アクティブウィンドウがChromeの場合、UI Automationを使用して「現在開いているページのURL」を正確に取得する。また、Chrome全タブのタイトル一覧も取得する（ウェブコンテンツ内のタブUIは除外）。

### バックグラウンド監視
最前面ではないが、デスクトップ上に「開いている（可視状態の）」すべてのウィンドウ一覧をプロセス名と共に取得する。

### 定時実行（ポーリング）
1分に1回（設定可能）のサイクルで上記データを取得し、ログファイルに書き出す。

### 設定ファイル（config.toml）
- ポーリング間隔（秒）
- ログ出力ディレクトリ
- 除外するプロセス名リスト（完全一致、大文字小文字無視）

### コマンドラインフラグ
- `--interval`: ポーリング間隔（設定ファイルより優先）
- `--config`: 設定ファイルパス（デフォルト: 実行ファイルと同階層の `config.toml`）
- `--logdir`: ログ出力ディレクトリ（設定ファイルより優先）

### ログファイル出力
- JSONL形式（1行1JSONレコード）で日付別ファイルに追記（`<log_dir>/YYYY-MM-DD.jsonl`）
- 日付が変わると自動的に新ファイルにローテーション

## 3. 非機能要件（性能・運用）

### 低負荷性能
- 常駐時のCPU使用率を極限まで抑える（Goのネイティブコンパイルを活用）
- メモリ使用量を数十MB以下に抑える

### ポータビリティ
外部ランタイム不要で、.exe ファイル単体で動作すること。

### 常駐性
Windowsのスタートアップまたはタスクスケジューラ、あるいはWindowsサービスとしてバックグラウンドで安定動作すること。

## 4. 技術スタック

| 役割 | ライブラリ / 技術 | 理由 |
|---|---|---|
| OS API呼出 | `golang.org/x/sys/windows` | Windows標準のWin32 APIを直接叩くためのデファクト |
| UI解析 (URL取得) | `github.com/go-ole/go-ole` | ChromeのURL取得に必須なCOM / UI Automationを扱うため |
| プロセス管理 | `github.com/shirou/gopsutil/v3` | PythonのpsutilのGo版。プロセス名取得が容易 |
| 設定ファイル | `github.com/BurntSushi/toml` | TOML形式の設定ファイル読み込み |
